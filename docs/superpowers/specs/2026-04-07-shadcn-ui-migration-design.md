# shadcn/ui Migration & Theme Setup

**Date:** 2026-04-07
**Status:** Draft
**Scope:** `packages/ui`, `packages/config`, `apps/web`, `apps/desktop`

## Summary

Replace the hand-copied shadcn components in `packages/ui` with proper shadcn CLI-managed components, apply the Luma/Green/Mist theme (preset `b3dRfX4mKO`), and upgrade the shared packages from Tailwind v3 to v4 to align with the consuming apps.

## Goals

- Single source of truth for the design system in `packages/ui`
- Proper shadcn CLI support (`components.json`) so future component additions are one command
- Consistent Luma/Green/Mist theme across `web` and `desktop` apps
- Resolve the Tailwind v3/v4 version mismatch between shared packages and apps
- No breaking changes to consuming app component code

## Non-Goals

- Changing `apps/www` — it keeps its own standalone marketing palette
- Pre-installing shadcn components speculatively — add only when needed
- Building custom composite components beyond what shadcn provides

## Theme Configuration

Source: shadcn preset `b3dRfX4mKO`

| Setting | Value |
|---------|-------|
| Style | Luma |
| Base Color | Mist |
| Theme | Green |
| Chart Color | Teal |
| Heading Font | Geist |
| Body Font | Inter |
| Icon Library | Lucide |
| Radius | 0.5rem (medium) |

Exact CSS variable values (HSL tokens for light/dark, sidebar, chart-1 through chart-5) will be generated by the shadcn CLI during `init` with these options selected.

### Token Flow

```
packages/ui/src/globals.css    (defines all CSS variables — single source of truth)
    |
    ├── apps/web/layout.tsx    (imports @ramcar/ui/globals.css)
    ├── apps/desktop           (imports @ramcar/ui/globals.css)
    |
    └── apps/www               (does NOT import — keeps its own globals.css)
```

### Font Handling

`packages/ui/globals.css` references font CSS variables (`--font-heading`, `--font-body`) but does not load font files. Each consuming app is responsible for loading the actual fonts:

- **`apps/web`**: Via `next/font/google` (Geist already loaded, Inter to be added)
- **`apps/desktop`**: Via CSS `@font-face` or bundled fonts (determined during implementation based on existing desktop font strategy)

## Tailwind v4 Upgrade

Scoped to `packages/ui` and `packages/config` only — both apps already run v4.

### `packages/config`

- `tailwind.preset.ts`: Remove. Its only content is an unused `brand` color scale, now replaced by shadcn theme tokens in `globals.css`. Update the `exports` map in `package.json` to drop the `./tailwind` entry.
- `package.json`: Bump `tailwindcss` from `^3.4.0` to `^4`

### `packages/ui`

- `tailwind.config.ts`: Remove. In Tailwind v4, configuration is CSS-based. The shadcn CLI manages theme tokens via `globals.css` and component-level classes directly.
- `package.json`: Bump `tailwindcss` from `^3.4.0` to `^4`, align `postcss`/`autoprefixer` deps

### Risk

Tailwind v3 to v4 can introduce subtle class name changes. Since `packages/ui` only has 3 small components using standard Tailwind utilities, the surface area is minimal. A visual check after migration confirms nothing broke.

## shadcn CLI Setup

### `components.json` in `packages/ui`

```
Style:         Luma
Base Color:    Mist
CSS Variables: yes
Path aliases:
  components:  src/components
  ui:          src/components/ui
  lib:         src/lib
  utils:       src/lib/utils
Tailwind CSS:  src/globals.css
```

### Component Migration

1. Delete existing files: `button.tsx`, `card.tsx`, `input.tsx`
2. Run `pnpx shadcn@latest init` in `packages/ui` — generates `components.json`, updates `globals.css` with Luma/Green/Mist theme tokens
3. Run `pnpx shadcn@latest add button card input` — re-adds components via CLI

The CLI-generated components have the same API surface:
- `Button` with `variant`/`size`/`asChild` props
- `Card` with `CardHeader`, `CardFooter`, `CardTitle`, `CardDescription`, `CardContent` sub-components
- `Input` with `InputProps`

The `src/index.ts` barrel exports remain unchanged.

### Future Workflow

```bash
cd packages/ui
pnpx shadcn@latest add dialog    # writes src/components/ui/dialog.tsx
# Then add export to src/index.ts
```

## Consumer App Changes

### `apps/web`

- **`tailwind.config.ts`**: Remove the `@ramcar/config/tailwind` shared preset import (the preset is being deleted). In Tailwind v4, the remaining config (content paths) can move to CSS or stay as a minimal config file.
- **`src/app/layout.tsx`**: Add Inter font via `next/font/google`. Wire `--font-heading` (Geist) and `--font-body` (Inter) CSS variables.
- **`src/app/globals.css`**: Remove duplicate token definitions (`--background`, `--foreground`, `@theme inline` color block). Keep only `@import "tailwindcss"` and app-specific overrides. Theme tokens come from `@ramcar/ui/globals.css` already imported in layout.
- **Component code**: No changes.

### `apps/desktop`

- **`tailwind.config.ts`**: Same as web — remove the shared preset import.
- Ensure renderer entry point imports `@ramcar/ui/globals.css` for theme tokens.
- Load Inter and Geist fonts (via CSS `@font-face` or bundled — determined during implementation).
- **Component code**: No changes.

### `apps/www`

- No changes. Does not import `@ramcar/ui/globals.css`. Keeps its own palette.

| desktop | None | Import shared globals | Remove shared preset import | Add Inter + Geist |

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Theme scope | `web` + `desktop` only | `www` has its own marketing identity |
| Tailwind upgrade timing | Now, bundled with this work | Apps already run v4; no point doing CSS work twice |
| Component install strategy | On-demand via CLI | YAGNI — add components as features need them |
| `components.json` location | `packages/ui` | Single shared package is the source for all UI components |
| Existing components | Replace via CLI | Same API surface, zero consumer breakage, proper CLI tracking |
